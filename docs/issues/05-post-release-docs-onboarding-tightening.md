# Title

Post-release docs/onboarding tightening based on first external contributor feedback

## Problem

The repository now has strong release docs, but first-wave external feedback may still expose friction in onboarding, command discoverability, or cross-doc navigation.

## Why It Matters

- Improves first-time contributor success rate.
- Reduces repetitive support questions.
- Keeps public-facing trust high as project visibility increases.

## Acceptance Criteria

- Collect and triage first-wave onboarding feedback/issues.
- Apply targeted doc fixes in:
  - `README.md`
  - `CONTRIBUTING.md`
  - `docs/RELEASING.md`
  - `docs/DOCS_INDEX.md`
- Keep docs contract checks green after updates.

## Non-Goals

- Rewriting docs architecture from scratch.
- Expanding scope into semantic/compiler/runtime changes.
- Marketing-driven edits that overstate stability.
